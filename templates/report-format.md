## Report Format Reference

Skills produce two artifacts: a JSON manifest section (system state) and a Markdown report (the human view). The Markdown report is what the instructional designer reads. This file defines the canonical structure every per-skill report follows so the designer learns the shape once.

### Output contract

- **JSON.** `.idstack/project.json` — the manifest. The skill writes its section using `bin/idstack-manifest-merge`.
- **Markdown.** `.idstack/reports/<skill>.md` — the report. The skill writes it using the `Write` tool.
- **Pointer.** The skill's manifest section sets `report_path` to the relative path of the Markdown so other skills (and `bin/idstack-status`) can find it.

Re-runs overwrite the report. The timeline at `.idstack/timeline.jsonl` carries the run history; the report file always reflects the most recent run.

### Voice and stance

idstack is a collaborator, not a builder. Every report should read like evidence-based feedback to a designer who already has work to defend. Three rules:

- **Observation, then evidence, then suggestion.** Never recommend without first showing what was seen and what the research says about it.
- **Suggest, don't direct.** "Consider…" / "Evidence suggests…" / "You may want to…" — not "You must…" / "Add this." The designer owns the course; idstack offers the read.
- **Cite every recommendation.** Every finding carries `[Domain-N] [Tier]`. If you can't cite it, it doesn't belong in a finding — move it to a "Limitations" or "Notes" section.

### Canonical structure

```markdown
# {{Skill Name}} Report

**Course:** {{project_name}}
**Generated:** {{ISO-8601 timestamp}}
**Skill:** /idstack:{{skill}}
**Mode:** {{mode if applicable, e.g. "audit-existing", "gap-fill"}}
**Run id:** {{optional: short id from timeline.jsonl}}

## Summary

{{2–3 sentences. The 30-second read for a designer scanning the file.
Lead with the most important thing the designer should know — a score, a
top finding, a confidence statement. Don't bury the lede.}}

{{Optional: one-line scoreboard if the skill produces a numeric output —
e.g., "Quality 62/100. Accessibility 71/100. Confidence 4/5."}}

## Findings

{{One block per finding. Order by severity (critical → warning → info)
then by impact within severity. Use the structure below.}}

### Finding {{id}}: {{short title}}                        [severity] [tier]

**What we saw.** {{Observation grounded in the designer's course or manifest:
"Module 4 has no rubric." / "Of 8 ILOs, 3 have no matching assessment."
Concrete; pointer to the artifact when relevant.}}

**What the evidence says.** {{One or two sentences from the literature
synthesis. Use plain language, not jargon.}} [Domain-N] [Tier]

**Why it matters.** {{Bridge: explicitly connect the observation to the
evidence to a learner-outcome consequence. This is where the report
earns its keep — without this paragraph the citation reads as decoration.}}

**Consider.** {{Collaborative recommendation. Phrased as a suggestion,
not a directive. May reference another skill ("Run /idstack:learning-objectives
to address the bidirectional gap.") or a concrete change.}}

---

{{Repeat per finding. Stable ids (`<dimension>-<n>`) so other skills, the
parent, and the designer can refer to findings deterministically across runs.}}

## Top recommendations

{{The 3–5 highest-impact moves, ordered. Each one carries its evidence
citation and points to the skill or section that would address it.
This section is what `bin/idstack-status` and `/idstack:pipeline` may
surface as a digest.}}

1. **{{Action}}** [Domain-N] [Tier] — {{one-line why; pointer to the
   skill/finding}}
2. ...

## Limitations

{{What this report didn't analyze, what's a proxy, what would change the
read if available. Examples: "Manifest contains structure, not learner-
facing content text — these are proxies." / "WebSearch unavailable, so
evidence currency was not verified."}}

## Next steps

{{One paragraph. What the designer might do next inside idstack — a
specific skill to run, a specific finding to address first, or "rerun
this skill after addressing X." Avoid generic advice.}}

---

*Generated by `/idstack:{{skill}}`. The system-readable manifest section is in `.idstack/project.json` under `{{section}}`.*
```

### Notes for skill authors

- **Order matters.** Summary first; findings before recommendations; limitations before next steps. A designer scanning the top 200 words should know the verdict.
- **Severity taxonomy.** `critical` (course will measurably fail learners) / `warning` (likely problem worth addressing) / `info` (worth knowing, not blocking).
- **Tier taxonomy** comes from `evidence/references.md` — `T1` (RCTs/meta-analyses) through `T5` (expert opinion).
- **Don't pad.** Three solid findings beat ten thin ones. The designer's time is the binding constraint.
- **Match the manifest.** Every finding in the report must correspond to a finding in the manifest's structured arrays (`findings`, `wcag_violations`, `flags`, etc.) so downstream skills can read them programmatically.
- **No raw JSON in the report.** Tables, prose, and lists only. The JSON lives in the manifest; the Markdown is for the human.
